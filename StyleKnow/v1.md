**开发者指令（System Prompt for Code LLM）**：
你现在是一个资深的 Python AI 架构师。请阅读以下基于 LangGraph 的多 Agent 系统架构文档。该系统运行在纯本地内存中（不连接外部数据库），重点在于验证 Agent 之间的协作流转与内存中 NumPy 向量的增量计算。请在后续的对话中，严格遵循本文档定义的数据结构和节点逻辑来生成 Python 代码。

## 一、 核心创新点与技术约束（Context & Constraints）

1. **无外挂数据库 (In-Memory Only)：** V1.0 阶段禁用外部数据库连接。所有长期记忆（衣物库、偏好向量）和短期记忆（对话流转）均使用 Python 内置结构（`dict`, `list`）和 `numpy` 数组实现。
2. **增量偏好蒸馏 (Incremental Preference Distillation)：** 用户的个性化风格不依赖全量历史重新计算，而是通过针对 `numpy` 向量的指数移动平均（EMA）公式进行局部加权更新。
3. **双重负反馈闭环 (Double Negative Feedback)：**
* **内部闭环：** 生成 Agent 与带有“傲娇/毒舌”人设的校验 Agent 之间的 LangGraph 条件路由循环（最多 3 次）。
* **外部闭环：** 用户的前端点击行为（点赞/踩）直接转化为标量权重，修改内存中的场景偏好向量。

## 二、 全局状态图定义 (Graph State Schema)

在 LangGraph 中，所有节点共享并修改同一个状态字典。请使用 `typing.TypedDict` 严格实现以下结构：

| 字段名 | 类型 | 描述 |
| --- | --- | --- |
| `user_query` | `str` | 用户的原始自然语言输入（例："明天去三亚玩，帮我搭配"） |

| `iterations` | `int` | 当前的内部循环次数，初始值为 0，上限为 3 |
| `final_outfit` | `str` | 最终输出给用户的穿搭方案 |

## 三、 内存数据库规约 (InMemory Wardrobe DB)

需要实现一个单例类 `InMemoryWardrobeDB`，使用 `numpy` 生成 128 维的随机向量来 Mock (模拟) 真实的 Embedding 数据。

### 1. 数据结构

* **`items_collection` (衣柜库):**
* 结构：`dict[str, dict]`。Key 为衣物 ID (如 "item_001")。
* Value 包含：`name` (名称), `category` (分类：上衣/裤子/鞋), `tags` (标签列表，如 ["通勤", "黑色", "基础款"]), `vector` (128维 numpy array)。


* **`preference_vectors` (场景偏好库):**
* 结构：`dict[str, numpy.ndarray]`。Key 为场景标识 (如 "commute", "vacation")。
* Value 为 128 维代表该场景用户偏好的 numpy array。



### 2. 核心算法要求 (需用代码实现)

* **向量检索 (Vector Search)：** 使用纯 `numpy` 实现余弦相似度 (Cosine Similarity) 函数。检索时，根据给定的基准向量，返回 `items_collection` 中相似度最高的 Top-K 件衣物。
* **盲盒连通率逻辑 (Blind Box Logic)：** 检索出的 Top-K 列表中，必须硬性包含至少 2 件带有 "基础款" (basic) 标签的衣物，以保证生成的搭配不会完全冲突。

## 四、 节点执行逻辑 (Node Specifications)

### Node 1: `parse_context_node`

* **输入：** `user_query`
* **逻辑：** 模拟调用大模型，提取出 JSON 格式的意图。不需要真实调用天气 API，代码中可直接硬编码 Mock 返回值（如：`{"scene": "commute", "weather": "25°C 晴", "days": 1}`）。
* **输出更新：** `parsed_intent`

### Node 2: `retrieve_memory_node`

* **输入：** `parsed_intent`
* **逻辑：**
1. 根据 intent 中的 `scene` 从 DB 中获取对应的 `preference_vector`。
2. 调用 DB 的余弦相似度检索方法，召回 Top-8 件衣物。


* **输出更新：** `retrieved_items`

### Node 3: `generate_outfit_node`

* **输入：** `parsed_intent`, `retrieved_items`, `critic_feedback`, `iterations`
* **逻辑：**
1. 组合 Prompt：包含天气、可选衣服列表。
2. **关键：** 如果 `critic_feedback` 不为空，必须在 Prompt 中加入：“注意！上次生成的方案被退回，原因是：[{critic_feedback}]，请务必修正。”
3. 调用 LLM（初期可用 Mock 字符串替代，跑通流程后再接入真实 API）生成穿搭建议。


* **输出更新：** `draft_outfit`, 并将 `iterations` 累加 1。

### Node 4: `critic_evaluate_node`

* **输入：** `draft_outfit`, `parsed_intent`
* **逻辑：**
1. 注入“傲娇/毒舌造型师”的 System Prompt。
2. 核心校验规则：是否有颜色冲突？是否符合当前温度？
3. 强制 LLM 输出固定 JSON 格式：`{"pass": true/false, "reason": "毒舌吐槽内容"}`。


* **输出更新：** 如果 pass 为 false，更新 `critic_feedback`；如果为 true，清空 `critic_feedback`。

### Node 5: `update_preference_node`

* **输入：** 外界传入的用户真实反馈信号（1 表示喜欢，-1 表示讨厌）。
* **逻辑：** 执行核心的 EMA 向量更新公式：
* `current_outfit_vector` = 本次被选中的所有衣物 vector 的平均值 (numpy mean)。
* `new_preference = 0.9 * old_preference + 0.1 * current_outfit_vector * feedback_sign`
* 将 `new_preference` 覆盖回 DB。


* **输出：** 打印日志验证向量已更新。

## 五、 路由控制图 (Graph Routing Logic)

在构建 LangGraph 的 `StateGraph` 时，需遵循以下连线（Edges）：

1. `START` -> `parse_context_node`
2. `parse_context_node` -> `retrieve_memory_node`
3. `retrieve_memory_node` -> `generate_outfit_node`
4. `generate_outfit_node` -> `critic_evaluate_node`
5. **条件路由 (Conditional Edge) 位于 `critic_evaluate_node` 之后：**
* 如果 `pass == false` 且 `iterations < 3` ---> 路由回 `generate_outfit_node`
* 如果 `pass == true` 或 `iterations >= 3` ---> 路由至 `END` (或者进入等待外部信号的态)
